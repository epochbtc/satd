# satd 0.5.0

> Pre-release — in progress · [Changelog](../../CHANGELOG.md)

satd 0.5.0 is the wallet-backend release (Theme A), headlined by **BIP 352
silent payments**: a node-side tweak index and two ways to consume it — stream
per-block tweaks and scan client-side (the scan key never leaves the device),
or register a scan key and let the node match for you. The index is opt-in
(`silentpaymentindex=1`, default off); nothing changes for a node that does not
enable it.

This file accumulates entries as the feature stack lands; it is not yet a cut
release.

## Highlights

- **Silent payments (BIP 352) — foundation.** A shared BIP 352 kernel now backs
  both the tweak index and the scan-key matcher, so a tweak the node stores and
  a tweak a wallet recomputes are identical by construction. Validated against
  the BIP 352 v1.1.0 test vectors.
- **[Bitcoin Core v28+ obfuscated block files are now readable](#storage--core-compatibility)**
  — the documented "reuse a Core `blocks/` directory" migration works against
  modern Core datadirs with no configuration.

## Silent Payments

The kernel lives in a new workspace-internal crate, `node-sp-index`, alongside
the schema codec, backfill-cursor, read trait, and config that the rest of the
stack builds on. It has no daemon wiring yet — enabling the index, serving it,
and matching against it arrive in the following changes.

- **BIP 352 kernel.** Input eligibility and public-key extraction (P2TR with
  NUMS exclusion, P2WPKH, P2SH-P2WPKH, P2PKH including malleated scriptSigs;
  uncompressed keys and future-SegWit spends correctly skipped), the public
  tweak `T = input_hash · A`, and the scan loop (labels, `K_max = 2323` bound).
  The only cryptographic dependency is `bitcoin::secp256k1`, already in the
  tree. Correctness is measured as parity with the official BIP 352 vectors,
  not self-consistency.
- **`sp_tweaks` row format.** One row per block from taproot activation upward,
  present even when empty so row-presence distinguishes "indexed, no eligible
  txs" from "not indexed". Each row embeds the hash of the block it describes,
  making it self-authenticating — readers verify identity without consulting
  the height→hash index.
- **Index write path.** `silentpaymentindex=1` (default off, always compiled —
  no cargo feature) makes `connect_block` stamp one tweak row per block
  at/above taproot activation, in the same atomic RocksDB batch as the
  chainstate, so a served row can never describe a partially-committed block.
  Reorg disconnects remove the row by height; `-reindex-chainstate` rebuilds
  the index from genesis via the normal connect path. Two Prometheus counters
  (`satd_spindex_rows_total`, `satd_spindex_row_removes_total`) report committed
  rows.
- **Deferred backfill.** Turning the index on for an existing datadir does not
  silently leave the pre-flag history unindexed: the node reports the index as
  *not synced* (a `sp_index.complete` marker, cleared atomically whenever a block
  connects with the index off), and `backfillindex silentpayment` walks every
  block from taproot activation to the tip to fill it. The walk reconstructs
  spent prev-output scripts from undo data (no temp column family), anchors to
  the chain tip captured at start, aborts cleanly and cleans up its stale rows if
  a reorg invalidates that anchor, and resumes from its last 1000-block
  checkpoint across a restart or `kill -9`. `pauseindex` / `resumeindex` /
  `cancelindex silentpayment` drive it; `getindexinfo` grows a `silentpayments`
  section (synced flag + live backfill state, cursor/snapshot heights, ETA), and
  a `satd_spindex_backfill_progress_ratio` gauge tracks progress. A
  from-genesis sync with the flag on needs no backfill — it is complete by
  construction. The serving surfaces (streaming `tweaks` category + JSON-RPC
  `getsilentpaymentblockdata`) land in a later change.
- **Streaming wire schema.** The Streaming Consumption API schema
  (`satd-events-proto`) gains the messages both silent-payment consumption modes
  ride on. Tier 1 (client-side scan, zero custody) adds a `tweaks` firehose
  category — **bit 8, and deliberately not part of the `categories=0` "all"
  default**, so a pre-existing subscriber never starts receiving tweak volume
  after an upgrade — carrying a per-block `BlockTweaks` body (`TweakEntry` =
  33-byte tweak + optional txid/max-value), plus `tweak_dust_limit` and
  `tweaks_only` subscription knobs for server-side dust filtering and compact
  entries. Tier 2 (scan-key watch, the convenience mode) adds a scan-key watch
  kind — `AddSilentPayments` / `RemoveSilentPayments` control messages, a
  `SetWatchSet.silent_payments` field for atomic whole-set replace, and a
  `SilentPaymentMatched` event that carries the tweak + output counter so the
  client derives the output key offline. The scan secret is a watch credential,
  never a spending key. All of it is additive: the schema version does not bump,
  and a node or client that ignores these surfaces is byte-identical to before.
  Live emit, index-backed replay, the server-side matcher, and the SDK typed
  events and examples arrive in the changes that follow.
- **Tier 1 serving — stream your tweaks.** The `tweaks` firehose is now live on
  the gRPC `Subscribe` stream. On each connected block the node emits a
  `BlockTweaks` (only while a tweak subscriber is attached, so an idle index
  costs nothing on the bus), and a `from_cursor` resume replays the missing
  blocks straight from the index. A **tweaks-only** subscription cold-syncs from
  taproot activation in a single subscription: because every row embeds the hash
  of the block it describes and the fast path is gated on the index being
  complete, that replay is exempt from the 10,000-block replay clamp — a phone
  can catch up ~240k mainnet blocks of tweak data without a height→hash round
  trip. A **mixed-category** subscription keeps the clamp so cursors stay
  ordered. Per-subscription `tweak_dust_limit` (drop entries below a value floor)
  and `tweaks_only` (33-byte tweak alone, no txid/max-value) filter both live and
  replayed events. Asking for `tweaks` on a node with the index off, or
  requesting replay while it is still backfilling, is refused in-band rather than
  served with holes. A read-only JSON-RPC fallback,
  `getsilentpaymentblockdata "blockhash" ( verbosity dust_limit )`, serves the
  same bytes for scripts and integrators not yet on an SDK. The WS/SSE carrier
  and the typed SDK helpers follow in later changes; the server-side scan-key
  matcher (Tier 2) does too.

- **Tier 2 scan-key watch — server-side matching (confirmed path).** A
  constrained client that would rather not scan every block itself can register
  a scan credential and let the node do the ECDH. On the `Watch` stream,
  `AddSilentPayments` (or an atomic `SetWatchSet.silent_payments` replace)
  registers up to 16 targets per connection — each a `(b_scan, B_spend)` pair
  plus optional labels — and the node emits a `SilentPaymentMatched` for every
  output paying one of them as blocks connect. The event carries the output's
  key, value, and the public tweak `T` with its BIP 352 counter `k`, so the
  wallet re-derives the full output key (and hence its spending key) **offline**
  from its own `b_scan` — the node never holds spend authority. Matching
  recomputes from the block and its undo data via the same kernel the index uses,
  so it works with `silentpaymentindex` off, and it does zero extra work on any
  block when no target is registered. The scan secret is disclosed to the node to
  run ECDH — the operator (or anyone who compromises the node) learns *which*
  outputs are yours but can never spend them — so it is held in-memory per
  connection only, wrapped in a zeroize-on-drop buffer, and never persisted or
  logged. The same messages are mirrored on the WS/SSE JSON surface. The typed
  SDK helpers follow in a later change.

- **Tier 2 scan-key watch — mempool (unconfirmed) matching.** A registered scan
  credential now matches payments the moment their transaction is accepted into
  the mempool, not only when it confirms: the node emits a `SilentPaymentMatched`
  with `confirmed = false` on acceptance, and re-emits the same match with
  `confirmed = true` when the transaction lands in a block — exactly the two-phase
  shape the `ScriptMatched` watch already uses. The obstacle was data, not math:
  the default mempool event path keeps only a scripthash and value per prevout,
  which cannot classify input types or extract pubkeys. Rather than make silent
  payments silently depend on an operator raising `streamprevoutmeta` to `full`,
  the node retains the resolved prevout `scriptPubKey`s on each mempool entry
  **only while at least one silent-payment watch is live** — the matcher and the
  mempool share a single lock-free gate counter, so with no scan key registered
  nothing extra is retained and the mempool path is byte-for-byte what it was
  without silent payments. Matching is best-effort, like every mempool watch: a
  scan key registered after a transaction was already admitted matches it only
  once it confirms (the confirmed path and `RescanBlocks` cover the gap), and an
  RBF-replaced or evicted transaction simply never reaches the `confirmed = true`
  re-emit. An unconfirmed match carries no resume cursor — it is not a durable
  chain position — so a reconnecting client resumes from the last confirmed
  event. Mirrored on the WS/SSE surface.

- **Tier 2 scan-key watch — index-accelerated rescan.** Registering a scan key
  and then running a bounded historical `RescanBlocks` is how a fresh wallet
  cold-syncs its silent-payment history. When the tweak index is enabled and
  complete, that rescan now reads each block's tweaks straight from the index
  instead of reconstructing them from undo data and re-running the per-tx
  elliptic-curve math — the same acceleration the `tweaks` firehose replay
  already enjoys. It is gated block by block on the stored row's embedded block
  hash matching the block being scanned; any row that is absent, mismatched, or
  from a still-incomplete index falls back to recomputing that block from undo.
  Because both paths run the identical matching kernel, the fast path can only
  ever change *how fast* a rescan finds payments, never *which* payments — a
  reorg-heavy history rescanned with acceleration on and off yields identical
  matches. With the index off, rescans recompute exactly as before.

- **Rust SDK — both silent-payment modes, typed.** The published
  `satd-events-client` SDK now covers silent payments end to end. Tier 1 (the
  recommended zero-custody mode): a `Categories::TWEAKS` subscription with
  optional `tweak_dust_limit` / `tweaks_only`, delivered as typed
  `Event::BlockTweaks`, plus a runnable `sp_light_scan.rs` example that does the
  client-side ECDH locally — the scan key never leaves the process. Tier 2 (the
  convenience mode): `add_silent_payments` / `remove_silent_payments` on both the
  raw `WatchHandle` and the reconnect-resilient `ResilientWatch` / `WatchSetBuilder`
  (which re-register scan keys automatically after a drop), delivered as typed
  `Event::SilentPaymentMatched`, plus an `sp_wallet.rs` example that re-derives
  each payment's spending key offline from the tweak and counter. The scan-key
  helpers use the crate's default `bitcoin` feature to derive a target's removal
  identity.

## Storage / Core compatibility

### Core v28+ XOR-obfuscated block files (`blocksxor`)

Bitcoin Core v28.0 (October 2024) started XOR-obfuscating the `blocks/*.dat`
payloads on disk by default, keyed by a random 8 bytes persisted in
`blocks/xor.dat`. satd previously read block files raw, so the migration path
in `CORE_DIFFERENCES.md` — keep Core's `blocks/` dir, `-reindex-chainstate`
into satd — silently failed against any datadir created by Core v28.0 or
later.

satd now speaks the same on-disk dialect:

- **An existing `blocks/xor.dat` key is honored automatically** on every
  read and write path (block fetch, `-reindex`, `-reindex-chainstate`,
  the block-file audit, `satd-chainstate-repair`). Pointing satd at a
  default modern Core `blocks/` directory needs no configuration.
- **Plaintext stays satd's native format**: a freshly initialized blocks
  dir gets the all-zero key (recorded in `xor.dat` for Core
  interoperability), and existing satd datadirs are untouched — the zero
  key short-circuits the XOR entirely, so the plaintext path costs
  nothing. This deliberately diverges from Core's default of obfuscating
  new dirs; set `blocksxor=1` to opt into a random key on a brand-new
  blocks dir.
- **`blocksxor` is now a real option** (previously recognized-but-ignored):
  unset honors whatever the dir carries; `1` matches Core's default
  behavior for fresh dirs; `0` demands plaintext and refuses to start on a
  dir whose stored key is nonzero (Core-parity fail-closed, since silently
  ignoring the key would corrupt every subsequent write).
- Reindex scans also terminate cleanly on Core's zero-preallocated file
  tails under a nonzero key, and a block read under a mismatched key fails
  with a clear error instead of returning garbage.

Restart-required; not hot-reloadable via SIGHUP.

## Logging

Two operator-facing log-hygiene changes; no configuration and no format change
to the structured (JSON) log:

- **Per-transaction mempool acceptance is now `debug`, not `info`.** Under
  normal load the node accepts a transaction per mempool entry, so the
  "Transaction accepted to mempool" / "RBF replacement accepted to mempool"
  lines dominated the default-level log. They now sit at `debug`; raise
  verbosity to see them.
- **ANSI color is auto-detected.** The text log colorized unconditionally, so
  escape sequences leaked into `journald` and any piped or redirected log.
  Color is now emitted only when stdout is a terminal and `NO_COLOR`
  (<https://no-color.org>) is unset, so captured logs stay clean. A real
  terminal is unchanged.

## Upgrade notes

No action required — the silent-payment index is off by default; with the flag
unset, a node's defaults and on-disk behavior are byte-identical to 0.4.0.

The `blocksxor` support is likewise a drop-in: existing satd datadirs remain
plaintext; the only on-disk change is a new 8-byte `blocks/xor.dat` file
(all-zero for existing dirs) written on first start, which Core v28+ also
understands. Note that a blocks dir *created* with `blocksxor=1` is not
readable by satd 0.4.x or earlier.
