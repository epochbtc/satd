# satd 0.3.2

> Released 2026-06-24 · git tag [`v0.3.2`](https://github.com/epochbtc/satd/releases/tag/v0.3.2) · [Changelog](../../CHANGELOG.md)

A single-fix consensus maintenance release on the 0.3.x line. It corrects how
the block-timestamp (median-time-past) check resolves a candidate block's
ancestors, fixing a fork-handling bug that could permanently stall a node behind
the tip. No breaking changes; defaults stay Bitcoin Core-compatible. Upgrading is
recommended for every node, and required for nodes following testnet4.

## Highlights

### Median-time-past now walks the candidate block's own ancestors

`check_timestamp` implements Bitcoin Core's
`block.GetBlockTime() > pindexPrev->GetMedianTimePast()` rule. It was resolving
the 11 ancestors that make up the median through the **active-chain height→hash
index** instead of walking the candidate block's own parent pointers. For a block
on a *competing* branch, the height index returns the active chain's blocks at
those heights — not the candidate's real ancestors — so the block's
median-time-past was computed against the wrong branch.

On testnet4 the 20-minute minimum-difficulty rule produces a timestamp sawtooth
in which a competing branch routinely carries timestamps **below** the current
chain tip. When a node's active tip was a stale fork, every canonical successor
block fell below the fork's median-time-past and was rejected `time-too-old`
*before* it could become a reorg candidate — so the reorg that would have rescued
the node was exactly what the check blocked. The result was a node that held a
full set of peers and a current header chain yet sat frozen many blocks behind
the tip, downloading nothing, and did not recover across restarts.

This was observed live on a testnet4 node, stuck for ~11 days on an abandoned
equal-work minimum-difficulty fork while headers advanced normally.

The fix walks the candidate's own `prev_blockhash` chain (Core's
`pindex->pprev`), exactly as the difficulty retarget walk-back already did, so the
median is always taken over the candidate's branch and is immune to active-chain
height-index gaps. Both the live block-accept path and the background replay path
are corrected.

Difficulty checks, header acceptance, and the steady-state (non-forked) timestamp
check are unaffected — on a linear extension the parent walk and the height index
resolve the same ancestors. Only the competing-branch case changes.

## Upgrade notes

- Drop-in replacement for 0.3.1: no configuration, RPC, CLI, or on-disk format
  changes.
- A node already wedged on a fork before upgrading will not reorg automatically
  on restart — the binary fix prevents *new* wedges but does not by itself pull a
  competing higher-work chain. Recover such a node by invalidating its stuck tip
  with `invalidateblock <stuck-tip-hash>`, which re-activates the best valid
  chain.
